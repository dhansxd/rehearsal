# Vertical TDD evidence

The implementation was built in two vertical RED/GREEN cycles on 2026-07-18.

## Core engine

RED: `python3 -m unittest tests.test_vertical -v` failed with
`ModuleNotFoundError: No module named 'rehearsal'` after the seven required
behavior tests were written.

First implementation run: six behaviors passed and apply verification failed
because Python bytecode cache files polluted the state snapshot. The snapshot
boundary was minimally corrected to exclude generated bytecode.

GREEN: all seven core behaviors passed: isolation and real consequences,
contract correction, violation detection, failed-test gating, stale approval,
exact apply/rollback, and input/path rejection.

## Public workflow

RED: `python3 -m unittest tests.test_app -v` failed with
`ModuleNotFoundError: No module named 'rehearsal.app'` after the complete public
flow test was written.

GREEN: reset, unsafe rehearsal, correction, safe rehearsal, verified approval,
receipt, and verified rollback passed through the controller. The browser server
and UI are a thin adapter over that tested controller.

## Browser approval regression

An independent browser run exposed a race: the safe render revealed the Approve
button while all controls were still disabled, then enabled it in `finally`.
Immediate click-on-visible automation lost the click and emitted no HTTP request.

RED: the UI ordering regression test failed because `render()` preceded
`busy(false)`. Direct HTTP handler tests established that approve and rollback
routing itself was correct. GREEN: controls are now enabled before the safe state
is rendered, so every visible approval control is already actionable.

## Static asset cache regression

A subsequent browser acceptance run correctly exercised the old JavaScript
because the local server supplied no cache policy and HTML used stable asset
URLs. RED: tests showed both missing version query strings and a missing
`Cache-Control` header. GREEN: HTML now references versioned CSS/JS URLs and all
static responses carry `Cache-Control: no-store`, preventing stale demo assets
across source updates.

## Security review hardening

RED/GREEN groups cover the review findings: runtime reset containment; exact,
bounded, monotonic contracts; canonical path and hash-identical preservation;
loopback bind plus Host/Origin gates; preview ID/digest binding and serialized
mutation; HEAD/index drift; verified recovery after failures and exceptions;
trusted-seed-only tests; and count/boolean-only consequence metadata. Each group
was observed failing against the prior implementation before its minimum fix.

Arbitrary untrusted repository execution is deliberately unsupported rather
than presented as solved by a worktree; the engine rejects it without the
trusted-seed flag.

## 3-day quality program

The accepted P0/P1 scope was implemented in focused RED/GREEN groups:

- Contract correctness RED: conflict, unknown fallback, and proof-label tests
  failed (two failures, one error). GREEN: 3/3 focused, then 34/34 full.
- Approval freshness RED: expiry, replacement, and replay tests failed 3/3.
  GREEN: 3/3 focused, then 37/37 full.
- HTTP boundary RED: nonce, length matrix, timeout/headers, error redaction, and
  browser bootstrap produced two failures and three errors. GREEN: 12/12
  focused, then 42/42 full.
- State/error/accessibility RED: all 3 UI contract tests failed. GREEN: 3/3
  focused and 15/15 HTTP/UI, then 45/45 full.
- Receipt RED: controller evidence and UI export tests failed 2/2. GREEN: 2/2
  focused, then 47/47 full.
- Progress RED: the named-stage test failed 1/1. GREEN: 1/1 focused, then 48/48
  full.
- Uncertain recovery wording RED: the API incorrectly returned unchanged;
  GREEN: the tri-state regression passed, then 49/49 full.

P1 crash journaling was not attempted because fully proving crash recovery and
a process lease would add material risk to green P0 behavior. It remains an
explicit post-hackathon limitation.
